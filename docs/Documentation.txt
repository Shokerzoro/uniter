

##### **0. Настройка проекта**
Это CMake проект на Qt C++ (17 стандарт), в QtCreator с гитом в корне. В нем есть 3 подпроекта: статическая библиотека Common, .exe для обновлений Updater и само приложение Uniter. Собирается в 2 вариантах с динамической линковкой (для debug и profile) и статической (RelWithDebInfo и Release). Использует GoogleTests для юнит-тестирования, которые собираются в отдельный .exe и запускаются при каждой сборке в отдельной консоли. Добавлен флаг --verbose для отображения вызовов компилятора.

Используются сторонние библиотеки - MuPDF для извлечения данных из pdf файлов, tinyxml2 для работы с XML. При этом они собраны в окружении MSYS с помощью MinGW, как и весь проект Uniter на Qt, что позволит производить кроссплатформенную сборку.

На серверной стороне используется Kafka как брокер сообщений с хранением этих сообщений в течении определенного времени, по истечении которого сообщения удаляются и клиент вынужден выполнять полную синхронизацию. 

В качестве хранилища тяжелых файлов используется MinIo, при этом сервер оперирует URL ссылками на файлы (в основном конструкторские, но также pdf стандарты и т.д.), а клиент запрашивает URL и токен для их загрузки по-требованию.

##### **1. Описание приложения**

Цель приложения - автоматизация учета материалов для производственных/проектных фирм в сфере машиностроения. Алгоритмы работы приложения используют преимущества ЕСКД, используя готовое КД для автоматического извлечения информации об изделиях, что включает структуру изделия, исполнения, используемые материалы. При этом автоматизация производится охватывает процессы: выпуск КД, закупки, производство (запасы + управление заданиями). 
В основе проектирования информационной системы находится предположение, что все возможности можно представить как CRUD операции над данными, в том числе и новые подсистемы. Это позволяет масштабировать приложение и добавлять поддержку других бизнес процессов.

В дальнейших планах: расширение производственного блока, возможность создавать цеха и маршрутизировать производство по ним, а так же читать технологическую документацию. Также является платформой для внедрения автоматизированного оформления чертежей с помощью ИИ в будущем. 
Также является платформой для внедрения кооперации внутри машиностроительной отрасли для публикования заказов по металлообработке.
##### **2. Архитектурные слои приложения**
Приложение организовано в четыре независимых слоя, каждый из которых отвечает за свою область ответственности.

###### 1. **Сетевой слой (Network Layer)**
Сетевой слой обеспечивает изоляцию всех внешних коммуникаций от остальных частей приложения. Состоит из трёх независимых компонентов, каждый из которых отвечает за свой канал связи. Все компоненты взаимодействуют с верхними слоями исключительно через AppManager(управление приложением), не имея прямых связей с бизнес-логикой или UI

###### **ServerConnector**
Канал общения с самописному серверу (движку авторизации и CRUD операций). Использует TCP/SSL соединение для защищённой передачи данных.

Ответственность:
- Установка и поддержание постоянного TCP/SSL соединения с самописным сервером
- Сериализация UniterMessage в XML и отправка на сервер
- Десериализация XML-ответов обратно в UniterMessage, "выплевывание" их наверх
- Буферизация исходящих CUD запросов при отсутствии соединения (для последующей отправки при восстановлении)
- Обработка таймаутов запросов (каждый запрос имеет лимит времени ожидания ответа)
Операции:
- Протокольные операции (MessageType = PROTOCOL) для аутентификации, запроса полной синхронизации, запрос на получения доступа к кафка, запрос на токены MinIO
- CRUD операции (MessageType = CRUD) - запросы от клиента на операции над ресурсами во всех подсистемах

###### **KafkaConnector**
Односторонний канал получения broadcast-уведомлений от распределённой системы сообщений Apache Kafka. Клиент выступает только в роли consumer (читатель), никогда не публикует сообщения.

Ответственность:

- Подключение к Kafka кластеру с использованием credentials, полученных от сервера через ServerConnector
- Подписка на топик компании, непрерывное чтение новых сообщений (только CRUD броадкаст уведомления), десериализация в UniterMessage и "выплевывание" наверх
- Запоминание последнего обработанного offset в OS Secure Storage с привязкой к конкретному User (для возможности продолжения чтения после перезапуска)
- Проверка доступности сохранённого offset при инициализации, выдача наверх ошибки для указания необходимости полной синхронизации (FULL_SYNC)

###### **MinioConnector**
HTTP-клиент для взаимодействия с объектным хранилищем MinIO S3 API. Обеспечивает операции загрузки и скачивания файлов (чертежей PDF, XML snapshots конструкторской документации).
Не имеет прямых credentials для доступа к MinIO. Вместо этого используется механизм pre-signed URLs: клиент запрашивает у сервера (через ServerConnector) временный URL с вшитым токеном доступа, действительный в течение короткого времени

Ответственность:
- Выполнение HTTP GET/PUT/DELETE запросов к MinIO по pre-signed URLs для загрузки/выгрузки файлов. Знает только атамарные задачи (какой файл скачать/загрузить и куда)


###### 2. **Слой управления приложением (Application Management Layer)**
**Компоненты**: AppManager, ConfigManager
**Ответственность**:
**AppManager**:
- Глобальная FSM приложения, управление жизненным циклом
- Обработка протокольных UniterMessage (например при инициализации)
- Маршрутизация UniterMessage между сетевым слоем и слоем данных
- Координация выполнения FULL_SYNC

**ConfigManager**:
- Парсинг User от сервера
- Генерация сигналов для создания виджетов-вкладок подсистем
- Управление при изменениях конфигурации


###### 3. **Слой управления данными (Data Management Layer)**
Слой управления данными обеспечивает абстракцию над хранилищем данных и файлами, предоставляя унифицированный API для верхних слоёв. Содержит два ключевых компонента: **DataManager** для работы с структурированными данными (реляционной базой данных) и **FileManager** для работы с файлами.

###### **DataManager**
Центральный компонент управления данными, обеспечивающий доступ к локальной базе данных и реактивное обновление UI через паттерн Observer.

**Ответственность**:
- Управление ресурсами (загрузка, обновление, удаление), работа с локальной базой данных.
- Прием CRUD операций от сетевого слоя через AppManager и их обработка (имплементация)
- Управление подписками UI-компонентов на изменения данных (уведомления)
- Сброс всех данных при выполнении FULL_SYNC по команде от AppManager, впоследствии пассивный прем CRUD операций

**Три типа подписок**:
1. **ResourceListObserver** — на список ресурсов (для генеративных подсистем)
2. **ResourceObserver** — на конкретный ресурс по ID (для динамических виджетов)
3. **TreeObserver** — на древовидную структуру (для иерархических данных)

Механизм подписки через SubscribeAdaptor:  
Для избежания проблем синхронизации, висячих указателей, броадкаст уведомлений и прямых вызовов между DataManager и Observer'ами используется класс-адаптер SubscribeAdaptor.

###### **FileManager**
Координатор всех операций с файлами, обеспечивающий централизованное управление загрузкой, скачиванием и кэшированием файлов из MinIO.

Ответственность:
- Координация цепочки операций по извлечению файла - от запроса **presignedUrl** от сервера к запросу файла к MinIO.
- Проверка целостности файлов через SHA256 hash (берет из DataManager)
- Управление подписчиками через IFileObserver (уведомление о выполненной загрузке/выгрузке).

Этот слой находится между Data Layer и UI Layer, используя DataManager и FileManager для доступа к данным, но инкапсулируя сложную логику обработки.

###### ****PDMManager****
Product Data Management Manager — подсистема управления конструкторской документацией и изделиями. Выполняет сложные задачи по восстановлению shapshot версий проектов, применение/отмена дельт. 

Ответственность:
- Взаимодействие с FileManager DataManager AppManager UI(предоставление API)
- Парсинг и генерация XML snapshot'ов конструкторской документации (КД)
- Управление версиями изделий (Part, Assembly, Project)
- Обработка delta-изменений (инкрементальные обновления snapshot'ов)
- Валидация КД на соответствие стандартам ЕСКД
- Связка изделий с чертежами (PDF файлы в MinIO)

###### ****ERPManager****
Подсистема аналитики материалов и производства, работает локально и занимается вычислением дефицитом/прогнозированием, используя данные по проектам, закупкам и т.д.

Ответственность:
- Расчёт потребности в материалах на основе производственных планов
- Прогнозирование дефицита материалов
- Анализ доступных запасов vs требуемых количеств
- Построение отчётов по материальному балансу

##### **3. Подсистемы**

Каждая подсистема владеет собственными ресурсами, которыми управляет с помощью CRUD операций. Ресурсы/подсистемы могут ссылаться на ресурсы из другой подсистемы с помощью id. Каждому ресурсу присваивается уникальное id, распределяемое централизовано. Каждый хост хранит все данные о всех ресурсах, а сервер лишь коммутирует CRUD операции.

**Подсистема базы материалов**
Управляет шаблонами материала MaterialTemplate, который отражает конкретный стандарт и его возможные варианты, служит основой для введения MaterialInstanse (независимая сущность, позволяющая указывать на стандарт и специализировать его). Внутри MaterialTemplate находятся векторы для их связывания (когда материал обозначается через дробь - сортамент/материал). Также владеет ресурсам MaterialTemplateComplex - это пара MaterialTemplate'ов, которые также могут являться основой для MaterialInstanseComplex. 

**Подсистема ссылок на материалы**
Тут производится управление ресурсами, которые напрямую завязаны на предыдущий раздел. Управляет собственными ресурсами MaterialInstanse, которые специализируют конкретный MaterialTemplate и добавляют к нему Quantity. В итоге формируется "экземпляр" материала, на который можно ссылаться например закупка, или деталь.

**Подсистема конструктора**
Управляет проектами project (независимый ресурс со своим id), которые внутри себя содержат сборки, детали и их документацию. Все эти сущности ссылаются на MaterialInstance - специализация шаблона материала. 

**Подсистема PDM**
Управляется snapshot (независимый ресурс со своим id), который внутри себя содержит слепок какого-либо проекта, и дельтами, которые являются ресурсом внутри определенного снэпшота. Подсистема работает поверх подсистемы конструктора и позволяет вводить управление версиями конструкторских проектов, предлагать изменения, а также принимать их либо отклонять.

**Подсистема закупок**
Управляет закупочными заявками ProcurementRequest, которые содержат MaterialInstance и комплексными заявками ProcurementRequestComplex, которые содержат внутри себя ссылки на простые закупки. 

**Подсистема менеджера**
Управляет особыми ресурсами - сотрудники, производства, интеграции, точно также с помощью CRUD операций. При этом производства являются генеративными ресурсами, т.е. после создания производства появляется подсистема этого производства, которая управляется через id, и управляет ресурсами внутри своей подсистемы.

**Генеративная подсистема производства X**
Управляет ресурсом ProdactionTask (отдельный ресурс со своим id) и также содержит список MaterialInstance. (PS: может выполнять запросы к данным из конструкторской подсистемы и к подсистеме закупок, что позволяет налету производить расчет дефицитов материалов).

**Генеративная подсистема интеграции X**
Управляет ресурсом - интеграции, которые позволяют открывать доступ к ресурсам (в первую очередь проектам) для других компаний. Использует ссылки на конструкторские проекты и материалы. Реализуется с помощью шлюза на сервере, при прохождении через который сообщение UniterMessage транслируется из одного пространства id ресурсов в другое, благодаря таблице соответствия шлюза. Сервер отслеживает, находится для id ресурса, к которому применяется операция CRUD в этой таблице, и если она там находится коммутирует сообщение. Пока что не реализуем.


##### **4. Проработка UniterMessage**

Сообщение UniterMessage выполняет функцию универсального контейнера для обмена данными между компонентами системы в трёх основных сценариях:
1. **CRUD операции между подсистемами** — пользователь на одном хосте создаёт/изменяет/удаляет ресурс (например, новое поступление в закупках), сервер распространяет это изменение всем клиентам компании через Kafka
2. **Протокольные операции с сервером** — аутентификация, запрос полной синхронизации (FULL_SYNC), получение credentials для Kafka и токенов для MinIO
3. **Операции с файловым хранилищем** — предназначен для MinIOConnector и выполнения запросов по загрузке/скачиванию файлов

UniterMessage является универсальным контейнером данных, который **не содержит специфичные структуры для каждого типа ресурса**. Вместо жёсткой типизации используется гибкая иерархическая структура данных через XML payload, что позволяет легко расширять систему новыми типами ресурсов без изменения протокола

###### **Структура UniterMessage**

**MessageType** — тип сообщения, определяет какие дополнительные поля используются:
- `CRUD` — операции создания/чтения/обновления/удаления ресурсов
- `PROTOCOL` — протокольные операции с сервером
- `MINIO` — операции с файловым хранилищем

**MessageStatus** — статус сообщения в цепочке запрос-ответ:
- `REQUEST` — запрос от клиента (исходящее сообщение)
- `RESPONSE` — ответ от сервера (входящее сообщение)
- `ERROR` — ошибка при обработке запроса
- `NOTIFICATION` — уведомление о изменении данных другим пользователем (из Kafka)
- `SUCCESS` — подтверждение успешного выполнения собственной CUD операции (из Kafka)

###### **Поля для CRUD операций (MessageType = CRUD)**

**Subsystem** — подсистема, к которой относится ресурс и конкрентый **ResourceType**:
- `MATERIALS` — управление материалами
- `INSTANCES` - управление элементами материалов
- `DESIGN` — конструкторская документация
- `PDM` - управление версионированием
- `PURCHASES` — закупки и поставки
- `MANAGER` — управление предприятием
- `GENERATIVE` — производственные подсистемы

**CrudAction** — тип операции:

- `CREATE` — создание нового ресурса
- `UPDATE` — обновление существующего ресурса
- `DELETE` — удаление ресурса
- `READ` — чтение ресурса (не используется в текущей архитектуре, все чтения идут из локальной БД)

И также некоторые общие поля, по типу id_пользователя, время создания, payload для передачи каких то данных (по типу логин/пароль).


**Объединение компаний**
При создании интеграции (подтверждение с обеих сторон) сервер это регистрирует. Интеграции как и другим ресурсам присваивается свой id, каждый свой для конкретной компании. После этого сервер создает шлюз пространств id, изначально пустой. Это просто таблица соответствия одних id к другой компании на по всем ресурсам. При при установки специального флага в UniterMessage требуемые ресурсы (в первую очередь проект) будут заносится в шлюз. Плюс сложный алгоритм разрешения остальных ресурсов, на которые проект ссылается. После этого при коммутации сообщений будут провертся, отмечены ли данные ресурсы в шлюзе, и переводится в пространство другой компании.


##### **5. Структуры для описания ресурсов**

Каждый общий ресурс (запись в базе данных материалов, закупка, проект, что-то еще) также владеет уникальным id в рамках своего типа. Получение нового id управляется также централизовано, через сервер. 

При этом ссылкой на общий ресурс является просто указание его id и тип ресурса. Но когда производится создание либо обновления ресурса, тогда необходимо передавать полную информацию.

Обновление ресурсов - без delta, всегда итоговый результат, что обеспечивает дополнительную надежность для повторного применения. Отображение ресурса решается на уровне структур данных ресурсов, а не на уровне сообщения. Выигрывает сообщение с большим id, у каждого клиента есть очередь сообщений, он применяет и подтверждает только следующий id.

###### **Материалы**
В подсистеме базы материалов производится управление шаблонами материалов. Шаблон материала моделирует стандарт, имеет вектора сегментов суффикса и префикса, которые обозначаются с помощью id. Каждый сегмент имеет свой набор значений, которые также имеют свои id. Удаление и добавление новых сегментов не ломает структуры, которые через id указывают на сегменты и их значения, т.к. каждый имеет свой уникальный id в своей области применения. Также шаблон содержит векторы, определяющие порядок применения сегментов.

enum class SegmentValueType : uint8_t { 
	
	STRING = 0, // Произвольная строка 
	ENUM = 1, // Выбор из списка 
	NUMBER = 2, // Число (int или double) 
	CODE = 3 // Код (специальный формат) 
};

    bool is_active = true; // true = активен, false = удалён (deprecated)

#### **7. Порядок рантайма**

1. Создание главного окна (статических виджетов). При этом устанавливается базовое состояние 
2. Создание менеджера данных, связывание сигналов/слотов с главным окном
3. Создание менеджера приложений, который создает менеджера конфигураций и локальных настроек. Связывание сигналов/слотов с главным окном и менеджером данных.
4. Создание сетевого класса, связывание сигналов/слотов с менеджером приложения.
5. Запуск менеджера приложений (короткий вызов функции которая только отправит сигнал и вернется в цикл обработки сигналов). При этом он отрабатывает FSM инициализации: отправляет сигнал сетевому на начало подключения, и основной поток входит в цикл обработки сообщений.

6. Сетевая инициализация
	1. Сетевой класс получает ip сервера через DNS
	2. Устанавливает SSL соединение, после чего отправляет сигнал signalConnected.
7. Авторизация и конфигурирование
	1. Менеджер приложения запрашивает данные для авторизации у виджета авторизации, и отправляет запрос серверу
	2. В случае успешной авторизации сервер присылает resources::User.
	3. Менеджер приложения инициирует менеджера данных, производится загрузка данных.
	4. Менеджер приложения передает User менеджеру конфигураций, что служит основой для генерации виджетов-вкладок
	5. Менеджер приложения запускает менеджера локальных настроек, который применяет локальные настройки, связывает локальные папки и проекты и т.п.
	6. Приложение переходит в состояние READY.
	7. Параллельно инициируется проверка и загрузка обновлений
8. Создание CRUD операций пользователем и прием сообщений
	1. Генерируется UniterMessage и отправляется на сервер
	2. После обработки сервером происходит рассылка UniterMessage
	3. Только после получения UniterMessage производится внесение изменений в локальную базу данных и изменение представления.
9. Создание новых генеративных вкладок
	1. Если CRUD операция затрагивает ресурсы, которые отвечают за генеративные подсистемы, сервер это обрабатывает как обычный ресурс.
	2. При добавлении для Users генеративной вкладки, серевер передает новый User клиентам.
10. Обработка изменения User 
	1. Получение нового User обрабатывается менеджером приложения как и прочие протокольные UniterMessage, и передается менеджеру конфигураций
 


#### **8. Коннект сигналов и слотов**

Динамические виджеты и виджеты подсистем используют IDataObserver для получения данных от локальной БД, подписки на какие то ресурсы, а также IFileObserver для доступа к файлам из сервера MinIO.

Также вкладки подсистем используют коннект с AppManager для отправки сообщений, а динамические виджеты используют подключение к своей подсистеме для отправки сообщений через нее.

Большинство объектов, которые существуют на протяжении всей программы реализованы как singltone, при этом коннектятся в main(), чтобы подчеркнуть некорректность произвольных коннектов в рантайме, кроме специально обозначеныых (только отправка сообщений для AppManager и авто-коннекты при создании обзервера). 

#### **9. Про PLM**
На входе у нас есть каталог с конструкторскими файлами. После парсинга мы переводим эти файлы в структурированный XML вид. Если парсинг производится в первый раз, тогда создается новый ресурс shapshot. Если второй, то на основе парсинга создается delta снэпшота (об этом ниже)

XML-структура снэпшота разделяет **ссылки** (структуру изделия) и **определения** (параметры элементов). Внутри каждой сборки есть два элемента **структура \<structure>** и используемые напрямую в этой сборке **детали \<parts>**. Внутри структуры содержится элемент **постоянных данных \<invariant>**, а также переменные данные для каждой **конфигурации \<config>**, которые суммарно дают полные данные о структуре для каждой конфигурации. При этом в структуре используются только ссылки на детали/сборки, например \<partref> \<assemblyref>. Полные данные о деталях содержатся внутри \<parts>.  Это исключает дублирование данных и позволяет переиспользовать детали в разных сборках.


	<shapshot id="" designation="" name="" version="2" previousVersion="1">
	
	  <Assembly designation="" name="">
	    <Metadata>...</Metadata>
	    <Documentation>
	      <File path="..." hash="sha256:..." modifiedAt="..."/>
	    </Documentation>
	    <Structure>
	      <Invariant>
	        <Assemblies><AssemblyRef designation="" config=""/></Assemblies>
	        <Parts><PartRef designation="" config=""/></Parts>
	        <StandardProducts/>
	        <BuyingProducts/>
	        <OtherMaterials/>
	      </Invariant>
	      <Config number="01">...</Config>
	      <Config number="02">...</Config>
	    </Structure>
	    <PartsDef>
	      <Part designation="" name="">
	        <Config id="01"/>
	        <Config id="02"/>
	      </Part>
	    </PartsDef>
	  </Assembly>
	
	  <Errors>
	    <Error severity="Error" category="FileSystem" type="NO_FILE"
	           path="drawings/4021.01.00.02.pdf"/>
	    <Error severity="Warning" category="VersionControl" type="INFORMAL_CHANGE"
	           designation="4021.01.00.03" hashBefore="sha256:a1b2..." hashAfter="sha256:c3d4..."/>
	  </Errors>
	
	</Shapshot>


А элемент partdef просто является контейнером для MaterialInstance + отстальная метаинформация. 


	<Partdef designation="4021.01.00.01" name="Балка продольная">
	  <Metadata>
	    <Material>

	      <CoatingBottom/>
	    </Material>
	    <Signatures>
	      <Signature role="Разработал" name="Иванов И.П." date="2026-01-15"/>
	      <Signature role="Проверил"   name="Петров С.А."  date="2026-01-18"/>
	    </Signatures>
	    <Litera>О</Litera>
	    <Organization>ОАО "Завод"</Organization>
	    <DrawingFile>
	      <Path>drawings/4021.01.00.01_рев.А.pdf</Path>
	      <Hash>sha256:e7f8g9...</Hash>
	      <ModifiedAt>2026-01-15T11:30:00+03:00</ModifiedAt>
	    </DrawingFile>
	  </Metadata>
	  <Config id="01">
	    <Dimensions length="2400" width="200" height="80"/>

	    <Dimensions length="2800" width="200" height="80"/>

	  </Config>
	</Partdef>


При этом по результатам парсинга производится отдельное управление ресурсами подсистемы конструктора, и отдельное управление ресурсами подсистемы PDM. Подсистема PDM владеет своими ресурсами - снапшотами проекта и дельтами внутри них, и позволяет управлять изменениями независимо, в т.ч. откатываться на предыдущие версии, предлагать изменения, принимать либо отклонять их.

###### **Управление версионностью (внутри формирования XML-дельты)**

Для управления версионностью используется подход логического отделения критериев изменений от данных, на основе которых производится решение о наличии изменений. Данные, на основе которых производится решение о наличии изменений, собираются независимо, в максимальном объеме, для всех критериев версионности. Сбор этих данных производится во время парсинга.

Вводятся критерии изменения документа - это значит что просто изменение хэша не будет являться достаточным основанием для фиксации изменений в XML дельта. Для фиксации изменений необходимо выполнение одного из критериев изменений (кроме очевидных добавлений конфигураций, или изменения габаритов, также оценивается изменение названия файла - например Рев.Б в конце, или добавление информации в основную надпись). 



#### **10. Архитектура Клиент-Сервер-Kafka-Postgres-MinIO**

Все ключевая информация хранится в реляционной БД (как на сервере так и локально). Все тяжелые файлы храняться в облаке MinIO, а если ресурс требует такого файла он хранит URL на них внутри БД. При этом клиент может запрашивать временный доступ к файлам через UniterMessage у сервера, а также закладывать туда файлы самостоятельно, например при создании ресурсов. Поля для запросов URL еще не проработаны в UniterMessage.


#### **11. Компилятор чертежей**

Алгоритм компиляции должен выглядить следующим образом. 

Сначала в идеале нам нужно получить список файлов, которые относятся к проекту. Но мы можем также и самостоятельно их определить по названию проекта и по именам файлов. Итак, мы сформировали инициирующий список pdf файлов.

Далее, т.к. в каждом pdf может находится несколько чертежей, мы формируем инициирующий список примитивов трансляции - каждая отдельная страница pdf, с сохраненным путем к нему и контекстом.

Далее нам нужно определить тип каждого примитива трансляции (спецификация, сборочный чертеж, деталь, монтажный чертеж, причем нужно отличать первую страницу от других). Для этого мы для каждой страницы извлекаем designation. Нам играет на руку то, что каждая первая страница содержит основную надпись одинакового размера в правом нижнем угле, а также designation в верхнем левом угле, а следующие страницы того же документа имеют тот же designation, только еще имеют запись листа, отличную от "1".

После этого мы уже трансформируем наш список из примитивов трансляции в единицы трансляции, логически объединяя документы. Далее производится распределение единиц трансляций по bucket-ам, с применением типо-специфичных компиляторов. 

После прохода типо-специфичных компиляторов производится уже линковка проекта в дерево. 


Важно! Создание XMLDocument, а также root элемента с его атрибутами выполняется на уровне PDMManager, а в библиотеку компиляции только передается root. При этом именно PDMManager управляет сохранением документа и т.д.
